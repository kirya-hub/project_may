# Задача: доработать приложение «Дроп недели» (app `drops`)

## Контекст проекта
Django-проект (loyalty/promo для кафе). Юзеры грузят чеки (`add_order.Order`), копят траты, получают купоны (`promo.CouponOffer` → выданный код `user_profile.PromoCode`). Приложение дропов уже частично написано — лежит в `drops/`. Страница доступна на `/drops/` (см. `config/urls.py` → `path('drops/', include('drops.urls'))`).

## ГЛАВНОЕ ПРАВИЛО — НЕ ПЕРЕПИСЫВАТЬ С НУЛЯ
Бэкенд дропов на ~80% уже есть в `drops/services.py` и `drops/models.py`. Твоя задача — **рефактор, а не новая реализация**. Не создавай параллельных функций расчёта лиги, выбора опций и т.д. Используй и правь существующие. Перед любыми изменениями прочитай целиком: `drops/models.py`, `drops/services.py`, `drops/views.py`, `drops/urls.py`, `drops/templates/drops/`, `promo/models.py`, `promo/services.py`, `user_profile/models.py` (модель `PromoCode`, `Profile`), `add_order/models.py` (модель `Order`), `add_order/views.py`.

Что уже работает и НЕ надо переписывать заново:
- `get_user_tier(user)` — расчёт лиги (GOLD / BRONZE / None) по тратам за неделю с множителем уровня.
- `ensure_week_options(user)` — создание `DropWeek` и 3 опций `DropOption` (рандомные купоны из БД по редкости и тиру).
- `choose_option(user, option_id)` — выбор 1 из 3.
- `try_complete_by_order(order)` — старая механика «сходи в кафе» (её убираем, см. ниже).
- Один дроп на юзера в неделю — гарантируется `UniqueConstraint(user, week_start)` в `DropWeek`.

## Целевая механика (что должно получиться)

### Таймлайн (важно — главная заковыка)
Выбор дропа открывается **только после конца недели**, а не во время неё. Каждая запись `DropWeek` живёт в трёх фазах относительно своего `week_start` (понедельник):
- **ACCUMULATE**: `[week_start, week_start + 7 дней)` — юзер копит траты. Опции НЕ показываются. Лига считается «вживую».
- **CLAIM**: `[week_start + 7, week_start + 14)` — окно выбора 7 дней. Если по итогам недели юзер попал в лигу — показываются 3 дропа, можно выбрать 1.
- **EXPIRED**: `>= week_start + 14` — не выбрал, дроп сгорел.

Поскольку время идёт, **одновременно живут две недели**: текущая (фаза ACCUMULATE) и прошлая (фаза CLAIM). Это норма. Реши через два хелпера:
- `current_week` = `DropWeek` для `week_start_for(today())` (фаза ACCUMULATE).
- `claimable_week` = `DropWeek` с `week_start = current_week_start - 7 дней`, если он в фазе CLAIM, заработал тир и ещё не выбран/не истёк.

### Лендинг `/drops/` (страница `drops_page`)
Сюда нужно перенести **дизайн из HTML-макета handoff от Claude Design** (файл `Drop Nedeli.html` / `Drop Nedeli bundle-src.html`). Это React-прототип — **воссоздай его визуально в Django-шаблоне** (`drops/templates/drops/drops_page.html`), не копируй React-структуру. Используй существующие `base.html` и `static/css/promo.css`-конвенции проекта. Ключевые элементы макета, которые надо отрисовать:
- Заголовок «Дроп недели» + подзаголовок.
- **Таймер** обратного отсчёта до конца ТЕКУЩЕЙ недели (`current_week.week_start + 7`). Зелёный пульсирующий индикатор «НЕДЕЛЯ АКТИВНА».
- **Карточки лиг** (Нет тира `<500₽` / Bronze `500–1999₽` / Gold `2000₽+`) — подсвечивай реальную текущую лигу юзера бейджем «ВАША ЛИГА» (из `get_user_tier`, вживую).
- **Текущая недельная сумма юзера** — выведи число, по которому считается лига. Подпиши явно: «с учётом множителя уровня» (т.к. это сумма × множитель за уровень, см. ниже). Юзер должен видеть прогресс.
- Таблица «Бонус уровня к сумме чеков» и таблицы шансов купонов Bronze / Gold (данные уже есть в `drops_info` view).
- Кнопка-CTA **«Забрать дроп»** внизу (sticky). Состояния:
  - **Серая/неактивная** — если у юзера нет тира на текущей неделе И нет `claimable_week`.
  - **Активная (горит)** — если есть `claimable_week` (можно выбирать прямо сейчас) ИЛИ если на текущей неделе уже есть тир (но забирать рано). Ведёт на отдельный URL (см. ниже).
- Если дроп за актуальный цикл уже выбран (`chosen_option` есть, статус COMPLETED) — на лендинге показать блок «дроп получен» + сам купон (используй существующий include `promo/includes/coupon_card.html`).

### Отдельная страница выбора — НОВЫЙ URL `/drops/grab/`
Создай новый url+view (напр. `drops:grab`), куда ведёт кнопка «Забрать дроп». Логика по состоянию:
- Есть `claimable_week` в фазе CLAIM со статусом CHOOSING → показать **3 карточки дропов** (по дизайну из существующего `drops_page.html`, блок CHOOSING — там уже есть стили карточек по редкости common/rare/legendary и кнопки «Выбрать»). Выбор → POST на `choose_drop` → купон выдаётся СРАЗУ (см. ниже) → редирект, показать выданный купон.
- Текущая неделя имеет тир, но ещё фаза ACCUMULATE → показать экран **«Дождись конца недели»** + отсчёт до `week_start + 7`. Выбор недоступен.
- Дроп уже выбран (COMPLETED) → показать «уже забрал» + купон.
- Нет тира и нечего забирать → сообщение «мало трат, не попал в лигу» (или редирект на лендинг).

## Конкретные изменения по файлам

### 1. `drops/services.py` → `get_user_tier`
- **Убрать** фильтр `total_sum__gte=200` — считать ВСЕ чеки за неделю. Дубли по-прежнему исключать (`is_duplicate=False`) — это фрод.
- **Оставить** множитель уровня (`adjusted = raw_sum * (1 + bonus/100)`) и пороги `>=2000 → GOLD`, `>=500 → BRONZE`, иначе `None`. Это и есть «сумма × число от уровня».
- Добавить **параметризованную версию** `get_user_tier_for_week(user, week_start)`, которая суммирует заказы строго за `[week_start, week_start + 7 дней)`. Нужна для расчёта ФИНАЛЬНОГО тира уже закрытой недели в фазе CLAIM (текущий `get_user_tier` всегда считает только текущую неделю — для закрытой недели он даст неверную сумму). `get_user_tier` оставь как обёртку над ней для текущей недели.
- Верни также само число `adjusted` (чтобы вывести на лендинге) — например, сделай функцию, возвращающую `(tier, adjusted_sum)`, или отдельный геттер.

### 2. `drops/services.py` → генерация опций и фазы
- Опции (`ensure_week_options`) генерировать **не во время недели, а при входе недели в фазу CLAIM** (date >= `week_start + 7`), и только если `get_user_tier_for_week` для этой недели != None. Редкости/купоны — по существующей логике (`_pick_rarity`, `_pick_reward_offer`), тир брать финальный для той недели.
- Добавь хелперы фаз: например `phase(week)` → `'ACCUMULATE' | 'CLAIM' | 'EXPIRED'` по датам; `claim_expires_at = week_start + 14 дней`. Обнови `_expire_if_needed`: EXPIRED ставится когда `today >= week_start + 14` (а не +7).
- Добавь резолверы `get_current_week(user)` и `get_claimable_week(user)` по правилам из раздела «Таймлайн».

### 3. `drops/services.py` → `choose_option` (выдача СРАЗУ, без кафе)
- Сейчас выбор только ставит статус ACTIVE, а купон выдаётся позже в `try_complete_by_order` (когда юзер сходил в кафе). **Убираем поход в кафе.** Перенеси выдачу купона прямо в `choose_option`:
  - При выборе опции сразу создавай `PromoCode` (origin=DROP) из `chosen.reward_offer`, как это делает сейчас `try_complete_by_order` (тот же код генерации, та же логика «погасить предыдущий активный drop-купон»).
  - **Срок купона** считается от момента ВЫБОРА: `expires_at = today + DROP_EXPIRATION_DAYS[rarity]` (оставь 5/7/14 по редкости COMMON/RARE/LEGENDARY).
  - Статус недели после выбора → сразу `COMPLETED` (не ACTIVE). FeedEvent о выборе оставь.
  - **Важно: купон по-прежнему привязан к кафе** (`CouponOffer.cafe`). Мы убираем только обязательную миссию-поход, а не связь купона с кафе. Не выпиливай `cafe` из `DropOption`/`CouponOffer`.

### 4. `add_order/views.py`
- Убрать вызов `try_complete_by_order(order)` (строка ~100) и его импорт. Саму функцию `try_complete_by_order` в `drops/services.py` удали как мёртвый код (после переноса логики выдачи в `choose_option`). Проверь, что `Order` нигде больше на неё не завязан.

### 5. `drops/views.py` + `drops/urls.py`
- Перепиши `drops_page` под новый лендинг (данные: текущий тир + adjusted-сумма, таймер до конца недели, таблицы, состояние CTA, выданный купон если есть).
- Добавь `grab` view + url `path('grab/', views.grab, name='grab')`. Кнопка «Забрать дроп» на лендинге ведёт на `drops:grab`.
- `choose_drop` оставь POST-only, но проверку статуса поправь под новую модель (выбор разрешён только для `claimable_week` в фазе CLAIM).
- `drops_info` можно оставить или влить таблицы прямо в лендинг — на твоё усмотрение, но не дублируй данные.

### 6. Что НЕ трогаем
- Gold-лига сохраняет редкость **LEGENDARY** (в коде так — у юзера 3 вида купонов). Не приводи к макету, где у Gold только common/rare.
- Логику весов редкостей по уровню (`_rarity_weights_for_profile`) не меняй.
- Множитель уровня в расчёте тира — оставить.

## Технические требования
- После изменения моделей/полей — сделай и примень миграции (`makemigrations drops` + `migrate`). Если меняешь статусы/поля `DropWeek` — продумай дефолты, чтобы существующие записи не сломались.
- Не ломай другие приложения (`feed`, `promo`, `add_order` зависят от drops через импорты — проверь).
- Покрой ключевую логику тестами в `drops/tests.py`: расчёт тира за конкретную неделю, фазы (ACCUMULATE/CLAIM/EXPIRED), выдача купона сразу при выборе, срок купона по редкости, недоступность выбора во время недели, EXPIRED после 7 дней окна.
- Стиль и язык комментариев — как в проекте (русский).

## Чек-лист приёмки
1. `/drops/` показывает дизайн из макета: таймер до конца недели, карточки лиг с подсветкой реальной лиги, недельную сумму с подписью «с учётом множителя уровня», таблицы бонусов и шансов.
2. Нет тира → кнопка «Забрать дроп» серая. Есть тир, но неделя ещё идёт → кнопка горит, ведёт на `/drops/grab/` с экраном «дождись конца недели».
3. Неделя кончилась, юзер был в лиге → на `/drops/grab/` 3 дропа на выбор.
4. Выбор дропа → купон выдаётся сразу (без похода в кафе), привязан к кафе, срок 5/7/14 дней по редкости от момента выбора, статус недели COMPLETED.
5. Один дроп на неделю; повторный заход → «уже забрал» + купон.
6. Не выбрал за 7 дней окна → EXPIRED, дроп сгорел.
7. `try_complete_by_order` удалён, `add_order` его не вызывает, ничего не сломалось.
8. Миграции применяются чисто, тесты зелёные.
